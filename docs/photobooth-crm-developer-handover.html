Photobooth CRM Developer Handover

Setup, deployment, environment configuration, Stripe integration, and payment workflow documentation.

Project: Photobooth CRM / MemoShot multi-tenant Laravel application

Prepared: 21 April 2026

1. Application Overview

This application is a Laravel and Vue multi-tenant CRM for photobooth businesses. Each tenant is represented by a workspace subdomain, for example acme.memoshot.com. Tenants manage bookings, invoices, packages, equipment, add-ons, discounts, leads, customers, campaigns, templates, subscriber groups, workspace settings, and tenant-owned Stripe payment settings.

The central domain also contains a super admin area at /admin. The super admin manages tenants, subscription plans, tenant access, platform subscription history, unpaid subscription charges, server utility commands, and the production .env editor.

Important Separation

Area Purpose Stripe Account Used Money Goes To
Tenant invoice payments Customer pays an invoice/installment for a booking. Tenant Stripe secret from Workspace Settings. Tenant/business owner.
Platform subscription payments Tenant pays MemoShot/platform for CRM subscription. Platform Stripe secret from .env. Platform owner.

2. Technology Stack

  • PHP/Laravel 13.x application.
  • Vue admin frontend compiled by Vite.
  • MySQL or SQLite depending on environment.
  • Mailpit/Mailhog for local mail testing; SMTP/log mailer for other environments.
  • Stripe Checkout and Payment Intents for payments.
  • Dompdf is installed for PDF generation inside the app.

6. Tenant Stripe Settings

Each tenant configures its own Stripe values under Settings > Workspace Settings. These tenant values are used only for customer invoice/installment payments.

Workspace Setting Purpose
Stripe Secret Key Tenant-owned Stripe secret key. Used to create Checkout sessions for customer invoice payments.
Tenant Stripe Webhook URL Read-only helper showing https://memoshot.com/stripe/webhook.
Stripe Webhook Secret Tenant-owned webhook signing secret from that tenant's Stripe Dashboard webhook endpoint.
Stripe Currency Three-letter currency for tenant checkout sessions, for example aud.

7. Stripe Setup

Tenant Customer Invoice Payments

  1. Tenant logs in to their own Stripe Dashboard.
  2. Create a webhook endpoint with URL https://memoshot.com/stripe/webhook.
  3. Select event checkout.session.completed.
  4. Copy the generated whsec_... signing secret.
  5. Paste it into Workspace Settings as the tenant Stripe Webhook Secret.
  6. Paste the tenant Stripe Secret Key into Workspace Settings.
  7. Set the tenant Stripe Currency.

Tenant invoice payments include invoice_id and installment_id metadata. The webhook verifies the tenant-specific webhook secret before marking the installment as paid.

Platform Subscription Payments

  1. Platform owner logs in to the platform Stripe account.
  2. Create a webhook endpoint with URL https://memoshot.com/platform/stripe/webhook.
  3. Select event checkout.session.completed.
  4. Copy the generated whsec_... signing secret.
  5. Set it in production .env as PLATFORM_STRIPE_WEBHOOK_SECRET.
  6. Set PLATFORM_STRIPE_SECRET to the platform Stripe secret key.

Platform subscription payments include tenant_subscription_charge_id metadata. They also save Stripe customer/payment method IDs for future off-session subscription billing.

Webhook Endpoints

Endpoint Stripe Source Event Secret Location
/stripe/webhook Tenant Stripe account checkout.session.completed Workspace Settings
/platform/stripe/webhook Platform Stripe account checkout.session.completed Production .env

8. Payment Workflows

Customer Invoice Payment Flow

  1. Admin creates or sends an invoice for a booking.
  2. Customer opens the invoice payment link or pays from the public invoice page.
  3. Application creates a Stripe Checkout session using the tenant's Stripe secret.
  4. Customer pays in Stripe Checkout.
  5. Stripe sends checkout.session.completed to /stripe/webhook.
  6. App verifies the tenant webhook secret, marks the installment paid, recalculates invoice amount paid/status, and updates booking status.
  7. As a fallback, the success return URL includes session_id, and the app verifies the Checkout session directly with the tenant's Stripe secret before marking paid.

Platform Subscription Payment Flow

  1. Super admin creates subscription plans.
  2. Tenant selects a monthly, quarterly, yearly, or weekly plan in Workspace Settings.
  3. Tenant clicks pay subscription.
  4. Application creates a platform Stripe Checkout session using PLATFORM_STRIPE_SECRET.
  5. Customer/payment method is saved in Stripe for future off-session billing. The app saves only Stripe IDs such as cus_... and pm_....
  6. Webhook or verified return URL marks the tenant subscription charge as paid.
  7. Scheduled billing creates future charges, sends reminders, charges due subscriptions, sends thank-you emails, and retries failures.
The application does not store card numbers, expiry dates, or CVC values. Those remain in Stripe.

Platform Subscription Emails

  • Reminder email: sent before the due date based on PLATFORM_SUBSCRIPTION_REMINDER_DAYS.
  • Paid email: sent after successful platform subscription payment.
  • Failed email: sent after a failed automatic deduction, with retry timing based on PLATFORM_SUBSCRIPTION_RETRY_DAYS.

11. Tenant Domains and Wildcard DNS

Tenants are resolved by subdomain. For dynamic tenants such as acme.memoshot.com, configure Hostinger/DNS to support wildcard subdomains.

  • Create wildcard DNS record: *.memoshot.com pointing to the hosting IP.
  • Ensure the web server routes wildcard subdomains to the same Laravel app.
  • Ensure SSL covers wildcard subdomains or individual tenant subdomains as required by the hosting provider.

15. Troubleshooting

Problem Likely Fix
Payment succeeds in Stripe but remains pending. Verify correct webhook endpoint, webhook secret, selected event, and run php artisan optimize:clear after changing .env.
Tenant invoice payment goes to wrong account. Confirm tenant Workspace Settings has its own Stripe secret. There should be no global tenant Stripe fallback.
Platform subscription auto-charge does not run. Confirm Hostinger cron calls schedule:run every minute and PLATFORM_SUBSCRIPTION_AUTO_CHARGE=true.
Automatic charge fails because no payment method exists. Tenant must complete one platform Checkout payment first so Stripe customer/payment method IDs are saved.
CSS/JS loads from localhost or Vite in production. Run npm run build, deploy public/build, and remove any stale public/hot.
Please provide a valid cache path. Ensure storage/framework/cache, storage/framework/views, and related storage directories exist and are writable.
